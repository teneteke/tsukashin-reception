## Overview

A local-first, offline-capable front desk settlement app that runs in the browser.
It records "who paid how much, for what, and how" each day at the front desk.
SaaS (HiTouch) is the single source of truth for member data, class/timeslot info,
and payment methods. This app caches that data locally and adds the settlement layer
on top. All local data lives in an in-browser SQLite database (sql.js + OPFS).

## Approach

### Tech Stack

- **View**: HTML + CSS + vanilla JS. No framework. Single `index.html` with 2 screens (show/hide).
- **DB**: sql.js (SQLite on WebAssembly) with OPFS persistence. All queries go through `js/db.js`.
- **PDF**: jsPDF with embedded Japanese font (Noto Sans JP) for receipt generation.
- **Sync**: fetch API (future) or CSV file picker (initial). Triggered manually via [Update] button. All SaaS data flows through `js/sync.js` which acts as the translation layer.
- **Offline**: Service Worker + PWA manifest. Hosted on GitHub Pages.
- **External libraries**: sql.js, jsPDF, encoding-japanese (vendored in T4.0 for Shift_JIS export and import encoding auto-detection). No other dependencies.

### DB Schema

**7 tables total.** All TEXT date columns use ISO 8601 format. All monetary amounts are INTEGER in yen (tax-included). Schema version tracked in settings for migrations.

**settings** — Key-value store for app configuration.
| Column | Type | Constraints |
|--------|------|-------------|
| key | TEXT | PK, NOT NULL |
| value | TEXT | NOT NULL |

Initial seed: `csv_encoding: "utf-8"`, `schema_version: "1"`, `receipt_issuer: "テニスラウンジつかしん"`, `receipt_issuer_address: ""`, `receipt_issuer_phone: ""`, `receipt_issuer_invoice_number: ""`, `last_member_sync: ""`. (Receipt numbering is derived at issue time from receipt_log MAX+1 for the current fiscal year; no counter is stored in settings.)

**members** — Member master. Cached from SaaS. Extended with local-only fields.
| Column | Type | Constraints |
|--------|------|-------------|
| id | TEXT | PK, NOT NULL |
| name | TEXT | NOT NULL |
| name_kana | TEXT | NULL |
| phone | TEXT | NULL |
| class | TEXT | NULL |
| timeslot | TEXT | NULL |
| is_temporary | INTEGER | NOT NULL, DEFAULT 0, CHECK (0 or 1) |
| memo | TEXT | NULL |
| created_at | TEXT | NOT NULL |
| updated_at | TEXT | NOT NULL |

- SaaS sync overwrites all fields except `memo` and `is_temporary` for existing members.
- `memo` is local-only, never overwritten by sync.
- `name_kana` for katakana search.
- `phone` for phone number search.
- `class` and `timeslot` are defaults from SaaS; can be overridden per visit in transactions table.
- INDEX on name_kana, phone for search.

**products** — Product master. All prices tax-included.
| Column | Type | Constraints |
|--------|------|-------------|
| code | TEXT | PK, NOT NULL |
| name | TEXT | NOT NULL |
| category | TEXT | NOT NULL |
| price | INTEGER | NOT NULL, CHECK (>= 0) |
| is_provisional | INTEGER | NOT NULL, DEFAULT 0, CHECK (0 or 1) |
| is_active | INTEGER | NOT NULL, DEFAULT 1, CHECK (0 or 1) |
| created_at | TEXT | NOT NULL |
| updated_at | TEXT | NOT NULL |

- `is_provisional`: marks products created on the fly at the front desk. Can be formalized later.
- Code scheme (provisional): 0xx = lessons, 01x = gut stringing, 02x = retail, 03x = other.
- Soft delete via is_active = 0.

**payment_methods** — Payment method master. Synced from SaaS.
| Column | Type | Constraints |
|--------|------|-------------|
| id | INTEGER | PK AUTOINCREMENT |
| name | TEXT | NOT NULL, UNIQUE |
| is_active | INTEGER | NOT NULL, DEFAULT 1, CHECK (0 or 1) |
| sort_order | INTEGER | NOT NULL, DEFAULT 0 |

- Provisional seed (until SaaS data available): Cash, PayPay, Card.
- On SaaS sync: overwrite list with SaaS data.

**transactions** — One row per person per day. The front-desk settlement record.
| Column | Type | Constraints |
|--------|------|-------------|
| id | INTEGER | PK AUTOINCREMENT |
| date | TEXT | NOT NULL (YYYY-MM-DD) |
| member_id | TEXT | NOT NULL |
| member_name_snapshot | TEXT | NOT NULL |
| class_override | TEXT | NULL |
| timeslot_override | TEXT | NULL |
| payment_method_id | INTEGER | NULL (FK → payment_methods.id) |
| is_attended | INTEGER | NOT NULL, DEFAULT 0, CHECK (0 or 1) |
| is_received | INTEGER | NOT NULL, DEFAULT 0, CHECK (0 or 1) |
| memo | TEXT | NULL |
| created_at | TEXT | NOT NULL |
| updated_at | TEXT | NOT NULL |

- UNIQUE(date, member_id): one row per person per day.
- `class_override` and `timeslot_override`: if set, override the member master defaults for this visit. NULL means use member master values.
- `is_attended`: front-desk attendance confirmation. Independent of payment.
- `is_received`: payment received confirmation. When toggled to 1, payment_method_id must be set.
- `memo`: per-visit note, separate from member memo.
- INDEX on date. INDEX on (date, member_id).

**transaction_items** — Line items per transaction.
| Column | Type | Constraints |
|--------|------|-------------|
| id | INTEGER | PK AUTOINCREMENT |
| transaction_id | INTEGER | NOT NULL, FK → transactions.id ON DELETE CASCADE |
| product_code | TEXT | NOT NULL |
| product_name_snapshot | TEXT | NOT NULL |
| price_snapshot | INTEGER | NOT NULL, CHECK (>= 0) |
| quantity | INTEGER | NOT NULL, DEFAULT 1, CHECK (> 0) |

- Same product code added to same transaction: increment quantity.
- Snapshots preserve data at time of sale.
- INDEX on transaction_id.

**receipt_log** — Record of issued receipts for audit trail and numbering.
| Column | Type | Constraints |
|--------|------|-------------|
| id | INTEGER | PK AUTOINCREMENT |
| receipt_number | TEXT | NOT NULL, UNIQUE |
| member_id | TEXT | NOT NULL |
| recipient_name | TEXT | NOT NULL |
| date_from | TEXT | NOT NULL |
| date_to | TEXT | NOT NULL |
| total_amount | INTEGER | NOT NULL |
| description | TEXT | NOT NULL |
| issuer | TEXT | NOT NULL |
| issued_at | TEXT | NOT NULL |

- Immutable log. Receipts once issued are never edited or deleted.
- `receipt_number` format: fiscal year prefix + sequential number (e.g., "2026-0001").
- `date_from` and `date_to`: the date range covered by this receipt.
- `description`: auto-generated "〇〇代として" from item categories.

### SaaS Sync Architecture

`js/sync.js` is the sole translation layer between SaaS and the local DB. No other module talks to SaaS directly.

**Sync trigger**: [Update] button on Screen 1.

**What gets synced (SaaS → Local)**:
- Member master: id, name, name_kana, phone, class, timeslot. Overwrites all non-temporary members. Preserves local `memo` field. Preserves temporary members.
- Payment methods: name, sort_order. Overwrites local list.
- (Future) Reservation list for today: list of member_ids expected today. Creates transaction rows.
- (Future) Product master: if SaaS has product codes.

**What gets synced (Local → SaaS)**:
- (Future) Today's settlement data. Depends on SaaS API capability.
- For now: CSV export is the manual sync method (Local → Excel → SaaS).

**Sync implementation (initial)**: CSV file picker. User exports from SaaS, imports CSV into this app. `js/sync.js` parses CSV and maps columns to local DB schema.

**Sync implementation (future)**: fetch API to SaaS endpoints. `js/sync.js` maps API response to same local DB schema. App code outside sync.js does not change.

**SaaS data unavailable**: If SaaS data cannot be obtained (no API, no CSV), fields from SaaS are stored as NULL. App continues to function — user can manually enter data or work with partial information. Error state is visible but not blocking.

**SaaS swap point**: When LAWN provides API specs, only `js/sync.js` needs modification. The rest of the app is insulated. PLAN.md should be updated with new tasks for API integration at that time.

### Data Integrity Rules

1. All DB writes are immediate. No "save" button. Every user action writes to SQLite and persists to OPFS.
2. Snapshots (member_name_snapshot, product_name_snapshot, price_snapshot) are written at creation time and never updated.
3. Product codes cannot be changed after creation (PK, referenced by transaction_items).
4. Member CSV/SaaS re-sync does NOT delete transactions. Transactions survive with snapshots.
5. Receipt log is append-only. No edits, no deletes.
6. DB import is full replacement with two-step confirmation.
7. "Today" = JST midnight boundary (Asia/Tokyo timezone, `new Date()` in JST).
8. Deleting a person from today's visitor list deletes their transaction row and cascades to transaction_items (ON DELETE CASCADE). This is for "added by mistake" only.

### Error Handling Policy

1. **DB write failure**: Inline error message near affected element. Console log for debugging.
2. **CSV parse error**: Summary dialog with row numbers and reasons. Import valid rows, skip invalid. Show imported/skipped counts.
3. **CSV encoding**: Import assumes UTF-8. Export offers UTF-8/Shift_JIS selection.
4. **OPFS persistence failure**: Persistent warning banner: "Data may not be saved. Export your DB immediately."
5. **Product code not found**: Inline "Code not found" message. Do not create item.
6. **Duplicate person in today's list**: Focus existing row, show notice.
7. **SaaS sync failure / data not available**: Show which data could not be loaded. App continues to function with available data. NULL fields displayed as empty.
8. **Concurrent browser tabs**: On init, check via BroadcastChannel API. If another tab is active, show warning and block writes in second tab.

### Receipt Specification

- **Format**: A4 PDF, portrait orientation.
- **Generated by**: jsPDF with embedded Noto Sans JP Regular font.
- **Content** (top to bottom):
  - Title: "領収書"
  - Receipt number (e.g., "No. 2026-0001")
  - Issue date
  - Recipient name (editable before issue, default = member name)
  - Amount in large text with "税込" notation
  - Description: auto-generated from item categories ("〇〇代として"). If multiple categories, comma-separated (e.g., "レッスン料、テニス用品代として").
  - Breakdown table: date, product name, unit price, quantity, subtotal — one row per item, grouped by date if multi-day range.
  - Total row at bottom of breakdown.
  - Issuer block: name, address, phone, invoice registration number (all from settings, optional fields left blank if not set).
- **Numbering**: Fiscal year (April start) + sequential. Stored in settings as next number. Logged in receipt_log on issue.
- **Trigger**: [発行・印刷] button on Screen 2. Respects the date range selected on Screen 2.
- **No receipt for ¥0**: Button disabled/hidden when total is ¥0.

### Phase 9: Performance improvements from code review

Addresses the Critical, High, and Medium findings from the 2026-04-16 code review. The root cause of the Critical group is that `dbRun` in `js/db.js` calls `saveToOPFS` after every write, so bulk sync operations serialize and rewrite the entire database once per row. The `withTransaction` API in T9.1 removes this amplification for every consumer. Implementation order is strict: T9.1 lands first because T9.2 through T9.4 and T9.9 depend on it; T9.5 through T9.8 are independent and can be ordered by value.

- [x] **T9.1** Transaction-aware batched OPFS save helper — `js/db.js`. Add a `withTransaction(asyncFn)` API that wraps `BEGIN TRANSACTION`, `COMMIT`, and `ROLLBACK`, and defers the OPFS persistence call until after commit. Inside the callback, `dbRun` must perform the SQL write but skip the OPFS save so bulk loops do not re-serialize and re-write the entire database per row. After a successful commit, `saveToOPFS` runs exactly once. A thrown error inside the callback triggers rollback and no OPFS write. Nested calls raise an error for now; a depth counter can replace this if a genuine nesting use case appears. Existing `dbRun` call sites continue to work unchanged because the OPFS-skip behavior only applies inside a transaction frame. **Done when**: a synthetic workload that inserts 100 rows inside one `withTransaction` block produces exactly one OPFS write and completes an order of magnitude faster than 100 sequential `dbRun` calls. ## Concerns

- [ ] **Screen 2 date pickers do double duty in Phase 7.** The same range is used for the Screen 3 history view and for the receipt issuance range. If reception staff find this ambiguous in practice, a follow-up task can move the pickers onto Screen 3 only and change the receipt flow to always cover today on Screen 2, with a separate range picker dialog for past-period receipts.
- [ ] **SaaS API specs unknown.** All sync is CSV-based initially. `js/sync.js` is the only file that changes when API becomes available. Column mappings in sync.js are provisional and marked with comments for easy adjustment.
- [ ] **SaaS product code scheme unknown.** Local codes (001, 010, etc.) are provisional. If SaaS has codes, re-seed via CSV import. App accepts any string as code — no code change needed, data only.
- [ ] **SaaS data structure for class/timeslot unknown.** Using free text fields. When SaaS structure is known, may need to convert to enum or lookup table. Migration handles schema change.
- [ ] **OPFS on Safari.** Safari OPFS support is limited. Primary target is Chrome on PC. If iPad becomes a target, T6.5 covers testing and fallback. Cross-browser data point recorded during T11.3 on 2026-04-16: Edge on macOS behaves identically to Chrome across all three persistence scenarios (browser-close-and-reopen, device restart, cache-only clear with site data retained), so on Chromium-family desktop browsers OPFS behavior is consistent. Safari remains untested.
- [ ] **jsPDF font size.** Noto Sans JP Regular TTF is ~4MB. Increases initial load. Consider subsetting if load time is an issue. Service Worker cache mitigates after first load.
- [ ] **CSV column mapping for SaaS exports.** Column names and order are provisional. `js/sync.js` has a mapping layer that's easy to adjust. First real CSV from HiTouch will determine actual mapping.
- [ ] **Reservation CSV scope.** Initial schema is member id plus optional date only. When SaaS reservations also expose class or timeslot overrides for the visit, `js/sync.js` needs a new branch that writes class_override and timeslot_override on the created transaction row.
- [ ] **Payment-method protection rule granularity.** T5.2 protects any method referenced by any row in transactions across all history. If this becomes too strict in practice (for example, a retired method that the SaaS wants to remove) relax to "referenced on or after a certain date". Revisit after the first real SaaS sync.
- [ ] **CSV kind detection via header row.** T5.4 routes files by matching their header row against the alias tables. If a SaaS export includes unexpected extra columns or localizes headers inconsistently, a file may be categorized as unknown. The alias tables stay easy to extend, but the first real CSV should be used to fill them in.
- [ ] **DB import file extension.** `.sqlite` and `.db` are accepted by the file input in T4.2. If a backup carries no extension or a different one, users must rename before import. Document in the in-app popover text.
- [x] **GitHub Pages HTTPS.** Required for Service Worker. Provided by default. No issue expected. Verified on 2026-04-16 during T13.5: the live deployment at `https://teneteke.github.io/tsukashin-reception/` enforces HTTPS at the edge (`https_enforced=true` in the Pages API response) and the Service Worker registers successfully under the subpath deployment because the Service Worker is cached with `./sw.js` and every CACHE_FILES entry uses `./` relative paths, so the scope resolves to `/tsukashin-reception/` as expected.
- [ ] **Shift_JIS export.** Requires encoding conversion in JS. Test thoroughly with Japanese Excel in T4.1.
- [ ] **Browser storage quota.** OPFS has generous limits but can be evicted. Daily CSV export as backup. OPFS warning banner as safety net.
- [ ] **Concurrent tabs.** BroadcastChannel API check in T1.1 prevents dual-write corruption.
- [ ] **Noto Sans JP base64 conversion blocks the main thread.** The current `arrayBufferToBase64` implementation in `js/receipt.js` runs on the main thread and blocks for roughly 100 to 300 milliseconds on a 1.5 megabyte font. Deferred from Phase 9 because the one-time cost on first receipt issuance is masked by the existing loading toast. Revisit if staff report a UI freeze on first issue per session. Alternatives: `FileReader.readAsDataURL` (async), a Web Worker, or shipping a pre-encoded base64 JS file so no conversion is needed at runtime.
- [ ] **Service Worker cache bumps have drifted outside PLAN.md tracking.** The `CACHE_NAME` value in `sw.js` is `tsukashin-v17` at the time Phase 9 began, but the most recent version explicitly recorded in a completion log entry is `tsukashin-v14` (T7.9). Three cache bumps (v15, v16, v17) happened between T7.9 and the start of Phase 9 without a corresponding task or completion-log entry. The code is still correct — versions are strictly monotonic — but the plan is no longer a reliable record of the cache history. Going forward, every cache-version change should land in a numbered task (or at minimum a log entry) to preserve the audit trail.
- [ ] **CSV parser uses per-character string concatenation.** `parseCsv` in `js/csv.js` builds fields by appending one character at a time. For CSV inputs beyond a few thousand rows this becomes noticeably slower than a chunk-based implementation. Typical SaaS member and reservation CSVs fit inside the current budget; revisit only if a large batch import exceeds the debounce or import-confirm dialog budget.
- [ ] **SaaS CSV kana column might be half-width katakana.** Phase 12 normalizes the search input side only; the stored form from sync stays as whatever the CSV provided. If HiTouch produces half-width katakana in the kana column, the T9.7 prefix LIKE will compare a full-width query against half-width stored text and miss. Decide the fix when the first real CSV arrives: either normalize on the storage side during `syncMembersFromCsv` (touches stored data, needs a migration for existing rows) or lift normalization into the SQL expression (no storage change, query-side only). The conservative default is the storage-side fix because it also benefits other places that read `name_kana`.
- [ ] **SaaS CSV kana column might be hiragana in rare cases.** Extremely unlikely for a Japanese gym SaaS but worth flagging. If it happens, Phase 12's input-side normalization is not enough on its own because the stored form is hiragana and the query is katakana after normalization. Handle through the same storage-side normalization path as the half-width concern above.
- [x] **Receipt number continuity (resolved in plan 2026-04-14).** No counter is stored in settings. Receipt numbers are derived at issue time from MAX(receipt_log) + 1 for the current fiscal year (April-start). A DB restore from backup is naturally consistent because the next number is always computed from the authoritative receipt_log. Implementation covered by T3.5.
- [x] **Visitor list event listeners were not wired up (resolved 2026-04-14).** initApp() in js/app.js now calls initVisitorListEvents() right after initEventListeners(), so the [+] popover, attendance/received toggles, inline class/timeslot selects, memo input, delete button, search bar, and sync-stub button all fire at runtime. navigateToList() was also updated to re-select the previously selected row on return from Screen 2.
- [x] **Service Worker cache list was missing js/visitor-list.js (resolved 2026-04-14).** sw.js CACHE_FILES now includes js/visitor-list.js; CACHE_NAME bumped from tsukashin-v2 to tsukashin-v3 to force a cache refresh on next activation.
- [ ] **Phase 14 receipt_log orphan edge case.** The receipt_log table references transactions by id but the foreign key relationship is not configured with cascade delete, so a transaction that is auto-deleted by the Phase 14 merge path would leave its receipt_log row dangling. In practice a receipt is only issued after a transaction is marked received, which engages the is_received protection rule and preserves the transaction from deletion. Verify during T14.3 that this invariant holds (by asserting receipt_log is always joined back to an existing transaction after merge) and add a pre-delete guard or a concern escalation if the invariant can be broken.
- [ ] **Phase 14 additive path removed.** Phase 14 replaces the existing additive sync rather than keeping both a merge variant and an additive variant. If a future need arises for a pure additive import (for example, first-time data seeding that should never delete local records), a flag parameter could restore the additive behavior without requiring a second function body. No action until the need is concrete.
- [ ] **Phase 14 post-merge dialog has no undo.** The summary dialog is read-only and staff cannot roll back an auto-deletion from the dialog itself. Recovery requires re-importing the prior CSV or restoring from the nightly SQLite export. If reception staff report this as a friction point, a follow-up task can snapshot the pre-merge state inside the merge transaction frame and offer an undo button in the dialog.
- [ ] **Phase 14 walk-in rule is defense in depth.** Walk-in member records are marked is_temporary equal to one by the app itself and are not part of the SaaS master. The SaaS CSV should never reference walk-in ids, so the walk-in arm of the protection rule normally does not fire. It is kept as a guard against a future CSV format that incorrectly exposes walk-in rows.
- [ ] **Phase 14 identity line formatting.** The summary dialog shows each row as a name-primary plus id-secondary pair. Confirm during T14.4 whether the design-system document defines a canonical member identity component that should be reused so styling stays consistent with other places where a member is named. If no canonical component exists, introduce a minimal utility class and record the choice as a design decision.
- [ ] **Phase 15 future-date transactions.** The working-date picker does not restrict the date to today or the past. Staff could set a future date and create transactions or walk-ins for a day that has not occurred yet. No guard is implemented because the expected usage is today or recent past days for corrections. If future-date entries become a problem, add a soft warning or a hard cap.
- [ ] **Phase 15 CSV sync on non-today working date.** When the working date is set to a past day, the CSV sync pipeline passes that date as the merge target. SaaS CSV exports typically represent today's reservations, so merging them against a past date would produce unexpected results. Staff awareness is sufficient for now; a future guard could warn when CSV sync is triggered while viewing a non-today date.
- [ ] **Phase 15 persist denial on Firefox.** Firefox uses engagement heuristics (notifications, bookmarks, frequent visits) to decide whether to grant persistent storage. A denial does not break the app but leaves OPFS in best-effort mode. The storage usage display from T15.2 shows the persistence status so staff and IT can see if it was granted.
